# 根据当前代码更新文档

以当前项目中的代码、测试和配置为依据，让指定文档保持准确、精炼且结构清晰。本项目需要维护的文档位于 `docs/ui/` 和 `docs/rules/`。

## 1. 确定范围

1. 读取仓库指令，以及 `docs/ui/` 和 `docs/rules/` 的入口文档。
2. 用户未指定范围时检查这两个目录；用户给出更明确范围时按其要求执行，并列出全部范围内文档。
3. 检查项目中的既有改动，避免覆盖用户工作。
4. 可以读取范围外代码和测试作为依据，但只修改用户授权的文档。

**完成标准：** 已得到明确的文档清单，并了解相关代码入口和既有改动。

## 2. 对照当前代码检查

逐份通读文档，并查阅相关代码、测试、配置和脚本，重点检查：

- 命令、路径、配置键、默认值、API 和入口是否仍然存在；
- 功能、UI、架构、约束和行为描述是否符合当前实现；
- 是否残留已完成的目标态、过时术语、旧流程或未实现能力；
- 文档之间是否不一致、重复、冗长，或缺少必要的当前事实；
- 文件职责是否重叠，是否确有合并或拆分价值。

源码中仅仅存在组件、类型或选项，不代表它已经接入产品。若仓库明确把某份文档定义为业务规则或契约，而代码与之冲突，则保留规则并报告冲突，不要直接把规则改成代码现状。

**完成标准：** 每份范围内文档都已对照相关实现检查，所有待处理项都有具体代码、测试或配置依据。

## 3. 更新并精炼

- 修正过时或错误内容，删除不再成立、重复或没有依据的描述。
- 相同职责的重复内容可以合并；职责、受众或用途不同的文件保持独立。
- 用简洁的当前事实替代实现过程和历史叙述，保留必要的架构、约定、命令、入口、约束和已知陷阱。
- 移动、合并或删除内容时，同步更新索引和链接。
- 只修改文档；发现代码问题时单独报告，除非用户同时授权修改代码。

**完成标准：** 已处理全部待办项，没有越出授权范围，文档结构和导航仍然完整。

## 4. 全文自检

修改后重新通读范围内的所有文档，包括未修改的文件，不要只看 diff。检查术语、命令、路径、能力描述和文档链接是否一致，并搜索本次替换的旧术语是否仍有残留。

运行仓库已有的文档或格式检查。若项目使用 Git，执行 `git diff --check`，再查看完整 diff 和项目状态。只有当文档结论依赖可执行行为时，才运行相关测试或运行态验证。

**完成标准：** 全部范围已复读，过时、不一致、重复和无依据内容均已处理；适用检查通过，剩余冲突或无法验证的内容已明确记录。

## 汇报

说明检查和更新了哪些文档、做了哪些合并或精简、运行了哪些验证，以及仍存在的代码冲突或验证缺口。未经用户要求，不执行 commit 或 push。
