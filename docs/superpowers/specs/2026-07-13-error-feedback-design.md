# 统一错误反馈设计

## 目标

统一 Web 前端的错误文案和呈现方式，避免将原始 `error` 直接插入页面。错误反馈应说明发生了什么、用户可以做什么，并在需要排障时提供默认折叠的技术详情。

本次只调整 `packages/web` 的用户界面与相关测试，不改变服务端业务规则或 API 契约。

## 设计原则

- 反馈层级由错误对当前任务的影响决定，不将所有错误统一为 Modal。
- 主文案使用中文和用户可理解的对象名称，保留必要的 `MCP`、`JSON` 等技术名词。
- 主文案不直接展示堆栈、HTTP 响应、内部路径、错误码或原始异常。
- 技术详情默认折叠，可包含经过脱敏的原始消息和错误码。
- 错误反馈给出明确的恢复路径；没有可执行操作时不添加无效按钮。
- 错误处理节点继续记录完整错误对象，不以界面文案替代日志。

## 统一模型

界面层使用统一的错误描述模型：

```ts
type AppErrorFeedback = {
  title: string
  message: string
  detail?: string
  code?: string
  action?: {
    label: string
    run: () => void | Promise<void>
  }
}
```

`normalizeErrorFeedback` 将 `ApiError`、普通 `Error`、已知错误码和未知异常转换为该模型。调用处可以提供面向具体任务的标题、回退文案和恢复动作。未知异常使用稳定的通用文案，原始值只进入经过脱敏的技术详情。

## 反馈层级

### FieldError

用于用户可以通过修改当前字段解决的问题，例如必填项、JSON 语法和变量值校验。错误紧贴对应控件，通过 `aria-describedby` 建立关联；用户修正字段后及时清除。

### ErrorToast

用于不阻断当前页面、可以重试或继续其他操作的失败，例如复制、保存、预览、刷新和后台请求失败。

- 显示标题、简短说明和关闭按钮。
- 可选地提供一个“重试”或同等明确的主操作。
- 存在操作时不自动关闭；无操作时使用较成功 Toast 更长的停留时间。
- 技术详情默认折叠，并可通过键盘展开。
- 全局宿主维护队列；短时间内相同错误合并并显示次数，避免通知堆积。

### ErrorDialog

只用于当前流程无法继续或需要用户作出决定的错误，例如删除、覆盖、冲突处理等流程的阻断失败。Dialog 打开后管理焦点，支持键盘操作和 `Escape`；普通请求失败不得默认升级为 Dialog。

### ErrorState

用于初始化失败、核心列表无法加载或局部编辑器无法工作等内容不可用状态。错误状态占据失效内容原本的区域，保留页面其余可用部分，并在具备恢复能力时提供“重试”。整个应用初始化失败时使用完整错误状态页。

## 文案规则

- 标题简短描述结果，例如“变量环境加载失败”。
- 说明提供下一步，例如“请检查网络后重试”。
- 操作名称与实际结果一致，例如按钮使用“重新连接”，成功反馈使用“已重新连接”。
- 不使用只有“出错了”“操作失败”或原始 `error.message` 的孤立文案。
- 已知错误码映射为稳定中文文案；例如 `session_expired` 显示为“MCP session 已过期，请重新连接”。
- 原始错误、错误码和诊断信息仅作为技术详情；任何密钥、token、变量秘密值和认证头必须在进入详情前移除。

## 数据流

1. 请求或本地操作失败后，错误处理节点记录完整错误对象。
2. 调用处将异常和当前任务上下文交给 `normalizeErrorFeedback`。
3. 字段校验直接进入 `FieldError`；其他错误按影响发送到 Toast、Dialog 或 ErrorState。
4. 用户执行恢复动作时，组件展示 pending 状态并防止重复提交。
5. 恢复成功后移除对应反馈并刷新内容；再次失败时更新现有反馈而非重复追加。

## 迁移范围

- `App` 初始化失败改为完整 `ErrorState`。
- Variables 的整页加载失败改为区域 `ErrorState`。
- Skills、MCP、Memory、Sync、Settings 和 Variables 的可恢复操作失败接入错误 Toast。
- 删除、覆盖和冲突处理等阻断流程按实际语义接入 `ErrorDialog`。
- JSON、必填项和变量值问题继续就地呈现，并统一为 `FieldError`。
- 编辑器自身加载失败保留为编辑器区域内的 `ErrorState`。
- 删除直接渲染未知 `error` 值的 JSX；已知诊断信息仍可在其所属上下文中展示。

## 可访问性与视觉规则

- Error Toast 使用适当的 live region；不会抢走当前输入焦点。
- Dialog 提供可访问名称、描述、初始焦点和焦点返回。
- FieldError 与控件建立程序化关联，颜色不是唯一错误标识。
- 所有关闭、展开和重试控件支持键盘操作并具有明确 accessible name。
- 动效遵守 `prefers-reduced-motion`；Toast 和 Dialog 不引发布局位移。
- 明暗主题均使用语义色 token，并保持正文对比度至少 4.5:1。

## 测试与验证

- 单元测试覆盖错误标准化、已知错误码映射、未知异常回退和敏感信息脱敏。
- 组件测试覆盖 Toast 队列、相同错误合并、自动关闭、详情展开和恢复操作。
- 组件测试覆盖 Dialog 焦点管理、`Escape` 和动作 pending 状态。
- 视图测试覆盖错误进入正确层级、重试成功后恢复内容以及字段修正后清除错误。
- 运行 Web Vitest 测试，并启动 `bun dev` 后使用 `playwright-cli` 验证桌面与窄屏交互、明暗主题和无重叠。

## 非目标

- 不修改服务端错误协议或业务规则。
- 不在本次改造中建立持久化通知中心或错误历史页面。
- 不向用户界面暴露完整堆栈或未经脱敏的响应体。
